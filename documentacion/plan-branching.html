1. Introducción

Este documento describe la estrategia de branching adoptada por el equipo de SmartLogix para organizar el trabajo en paralelo, controlar las versiones del software y garantizar que el código en producción siempre sea estable.

La estrategia elegida es GitFlow adaptado, basada en el modelo propuesto por Vincent Driessen (2010), con ajustes para equipos pequeños que trabajan en sprints cortos. Combina ramas de larga duración (main, develop) con ramas de corta duración (feature, release, hotfix).

Principio guía: La rama main solo contiene código que ya fue probado y aprobado. Nadie hace push directo a main ni a develop.

2. Diagrama General de Ramas

tiempo → main ●────────────────────────────●────────────────────● v1.0.0 ↑ v1.1.0 ↑ v1.1.1 │ │ release ────────────────●───────────● ●─────● v1.1.0-rc ────┘ (QA) └──→main hotfix └──→main ↑ ↑ develop ●────────●────●────────────────────────────────●────●──→ │ ↑ ↑ ↑ ↑ │ │ │ │ │ feat/inv─●──────→─┘ │ │ │ feat/ord──────────●──→┘ │ │ feat/pay──────────────────────────────────────────────●──→┘ │ feat/ship──────────────────────────────────────────────────●──→┘

Cada desarrollador trabaja en su propia rama feature. Al terminar, abre un Pull Request hacia develop. Cuando se acumula un conjunto de features probadas, se crea una rama release para QA final antes de fusionar con main.

3. Tipos de Ramas

main Rama permanente — Producción
Propósito:Contiene el código que está o estuvo en producción. Cada commit representa una versión liberada. Protegida:Sí. Solo acepta merges desde release/* y hotfix/* mediante PR aprobado. Etiquetas:Cada merge lleva un tag de versión semántica: v1.0.0, v1.1.0, etc. Push directo:Prohibido para todos los integrantes.
develop Rama permanente — Integración
Propósito:Base de integración donde se acumulan las features terminadas. Representa el estado del próximo release. Protegida:Sí. Solo acepta merges desde feature/* mediante PR con al menos 1 aprobación. Tests:Se ejecutan automáticamente al recibir un PR (CI/CD). Push directo:Prohibido.
feature/nombre-corto Rama temporal — Desarrollo de funcionalidad
Propósito:Desarrollo de una nueva funcionalidad o mejora. Una rama por tarea o ticket. Origen:Se crea desde develop. Destino:Se fusiona hacia develop mediante PR. Duración:Corta. Máximo la duración del sprint (1-2 semanas). Ejemplos: feature/inventory-stock-alert
feature/orders-payment-confirm
feature/shipping-tracking-ui
release/v1.x.0 Rama temporal — Preparación de versión
Propósito:Estabilizar el código antes de liberar. Solo se permiten bugfixes menores y ajustes de versión. Origen:Se crea desde develop. Destino:Se fusiona hacia main Y hacia develop (para sincronizar los fixes). Ejemplos:release/v1.1.0, release/v1.2.0
hotfix/descripcion Rama temporal — Corrección urgente en producción
Propósito:Corregir un bug crítico en producción sin esperar el próximo ciclo de release. Origen:Se crea desde main. Destino:Se fusiona hacia main Y hacia develop. Ejemplos:hotfix/gateway-502-dns, hotfix/payment-timeout

5. Flujo de Trabajo Detallado

5.1 Desarrollar una nueva funcionalidad

Crear la rama desde develop git checkout develop && git pull origin develop
git checkout -b feature/inventory-warehouse-delete
Desarrollar y hacer commits frecuentes git add src/controllers/inventoryController.js
git commit -m "feat(inventory): add warehouse delete endpoint"
Subir la rama al repositorio remoto git push origin feature/inventory-warehouse-delete
Abrir Pull Request hacia develop en GitHub Descripción del PR: qué hace, cómo probarlo, capturas si es UI. Asignar al menos 1 revisor del equipo.
Revisión de código y aprobación El revisor aprueba o solicita cambios. Se resuelven los comentarios.
Merge con Squash hacia develop Se usa Squash and merge para mantener el historial de develop limpio. Se elimina la rama feature después del merge.

5.2 Crear un Release

Crear rama release desde develop git checkout -b release/v1.1.0 develop
Actualizar versión en package.json de cada servicio npm version minor --no-git-tag-version (en cada servicio)
Pruebas de regresión y correcciones menores Solo bugfixes. No se agregan features nuevas en esta rama.
Merge hacia main y etiqueta git checkout main && git merge --no-ff release/v1.1.0
git tag -a v1.1.0 -m "Release v1.1.0"
Sincronizar develop con los fixes del release git checkout develop && git merge --no-ff release/v1.1.0

6. Convención de Commits

Se utiliza Conventional Commits para mantener un historial legible y poder generar changelogs automáticos.

PrefijoCuándo usarloEjemplo
feat:Nueva funcionalidadfeat(orders): add payment confirmation flow
fix:Corrección de bugfix(gateway): resolve DNS cache on container restart
docs:Solo documentacióndocs(readme): add Docker setup instructions
refactor:Refactorización sin cambio funcionalrefactor(inventory): extract stock logic to repository
test:Agregar o modificar teststest(bff): add integration test for products endpoint
chore:Tareas de mantenimientochore: update dependencies to latest minor
ci:Cambios en CI/CDci: add SonarQube quality gate step
Formato completo: tipo(scope): descripción en imperativo, minúscula, sin punto final
El scope es opcional pero recomendado: indica el servicio o módulo afectado.

8. Estructura de Repositorios

SmartLogix utiliza una estrategia de multi-repositorio: cada componente tiene su propio repositorio en GitHub, permitiendo que cada equipo despliegue de forma independiente.

RepositorioComponenteRama principal

smartlogix-shippingMicroservicio envíosmain
smartlogix-paymentMicroservicio pagosmain

Repositorio principal: Existe un repositorio raíz smartlogix que contiene el docker-compose.yml y la documentación general del proyecto. Todos los repositorios de servicios son submódulos Git de este repositorio raíz.

La combinación de Pull Requests, Conventional Commits y Semantic Versioning otorga trazabilidad completa del historial de cambios.
